# AerialClaw 3.0 — 多机协同架构设计方案

> 代号：v3.0 | 状态：设计阶段 | 日期：2026-04-02

---

## 一、版本定位

| 版本 | 核心能力 | 定位 |
|------|---------|------|
| v1.0 | 单机决策循环 + 硬技能 + Web UI | 能飞 |
| v2.0 | 安全包线 + 四层记忆 + 软技能 + 反思引擎 | 能学 |
| **v3.0** | **多机协同 + Agent 内核提纯 + 分层展示** | **能协作** |

v3.0 的核心目标：**让 N 架异构设备在统一架构下协同完成任务，同时让每一层的决策过程都可视化。**

## 三、核心改造：Agent 内核提纯

### 现状问题

当前 `server.py` 是 2300 行大杂烩：Flask 路由、Socket.IO 事件、Agent 初始化、技能执行、UI 状态管理全混在一起。多机架构下，主节点和子节点不需要摄像头 API，无人机不需要调度 API。必须拆开。

### 目标结构

```
aerialclaw/                        # Agent 内核（纯 Python，无 Web 依赖）
├── core/
│   ├── agent.py                   # AerialClaw Agent 核心类
│   │                              #   - 加载身份文档（SOUL/BODY/MEMORY/SKILLS）
│   │                              #   - 初始化技能注册表
│   │                              #   - 暴露 execute(instruction) → report 接口
│   ├── identity.py                # 身份文档读写
│   └── config.py                  # 配置管理
│
├── brain/                         # 认知决策层（现有，不动）
│   ├── agent_loop.py
│   ├── planner_agent.py
│   └── chat_mode.py
│
├── skills/                        # 技能体系（现有，不动）
├── memory/                        # 记忆系统（现有，不动）
├── perception/                    # 感知系统（现有，不动）
├── adapters/                      # 硬件适配（现有，不动）
│
├── swarm/                         # 多机协同层
│   ├── protocol.py                # 通信协议（已完成）
│   ├── node.py                    # 节点通信服务（已完成）
│   ├── commander.py               # 主节点逻辑（已完成）
│   ├── coordinator.py             # 子节点逻辑（已完成）
│   ├── executor.py                # 执行节点封装（已完成）
│   └── profiles/                  # 角色身份模板
│
└── interfaces/                    # 展示界面层（新增）
    ├── executor_server.py         # 单机控制台服务（从 server.py 拆出）
    ├── coordinator_server.py      # 子节点面板服务
    ├── commander_server.py        # 主节点大屏服务
    └── ui/
        ├── executor/              # 单机控制台前端（现有 UI 迁移）
        ├── coordinator/           # 子节点面板前端（新开发）
        └── commander/             # 主节点大屏前端（新开发）
```

### 关键原则

1. **Agent 内核零 Web 依赖** — `core/agent.py` 只依赖 Python 标准库 + LLM 客户端，可以在任何环境运行
2. **接口层可选** — 不启动 Web 服务也能通过 Python API 或 CLI 使用
3. **每种角色独立的界面服务** — 按需启动，不加载无关组件

## 五、通信架构

### 节点间通信

```
Commander ←── REST + WS ──→ Coordinator ←── REST + WS ──→ Executor
  :6000                       :6100                        :6200+
```

### 消息类型

| 消息 | 方向 | 用途 |
|------|------|------|
| REGISTER | 下→上 | 节点注册，上报身份和能力 |
| HEARTBEAT | 下→上 | 心跳保活，携带简要状态 |
| TASK_ASSIGN | 上→下 | 分配任务（自然语言指令） |
| TASK_STATUS | 下→上 | 上报任务进度 |
| TASK_REPORT | 下→上 | 提交任务报告 |
| QUERY_STATUS | 上→下 | 查询下级状态 |

### UI 数据推送

每层 Web 服务通过 Socket.IO 向前端推送实时数据：

| 数据 | Commander UI | Coordinator UI | Executor UI |
|------|-------------|---------------|-------------|
| 全局地图（所有设备位置） | ✅ | — | — |
| 区域地图（管辖设备位置） | — | ✅ | — |
| 任务分解/流转过程 | ✅ | ✅ | — |
| 摄像头画面 | — | — | ✅ |
| LiDAR 数据 | — | — | ✅ |
| 飞行状态 | 汇总 | 汇总 | 详细 |
| LLM 决策日志 | 本层 | 本层 | 本层 |
| 报告 | 全局报告 | 区域报告 | 单机报告 |

## 六、任务流转全流程

以"城市搜救"为例：

```
用户输入（主节点大屏）：
  "在城市搜救区域寻找被困人员，搜索完毕后提交报告"

Step 1 — Commander
  │  LLM 读取子节点能力：coord-A 管 2 架 UAV，coord-B 管 2 架 UAV
  │  LLM 分解任务：
  │    → coord-A: "搜索北部区域（建筑废墟带），重点关注倒塌建筑"
  │    → coord-B: "搜索南部区域（道路和开阔地），重点关注道路两侧"
  ▼
Step 2 — Coordinator-A
  │  LLM 读取下属能力：A1 有 LiDAR + 5 路摄像头，A2 仅有摄像头
  │  LLM 分配：
  │    → UAV-A1: "飞往废墟区 [40,-30]，用 LiDAR 扫描，搜索被困人员"
  │    → UAV-A2: "飞往 [60,-50]，用摄像头观察建筑外部，标记可疑位置"
  ▼
Step 3 — UAV-A1（现有 AerialClaw AgentLoop）
  │  接收自然语言指令 → 自主决策执行
  │    → 起飞 → 飞往目标 → 观察 → 检测 → 标记 → 生成报告 → 返回
  │  单机报告："在 [43,-33] 发现疑似被困人员信号，已标记"
  ▼
Step 4 — 报告逐级汇聚
  │  UAV-A1 + UAV-A2 报告 → Coordinator-A LLM 融合 区域报告
  │  UAV-B1 + UAV-B2 报告 → Coordinator-B LLM 融合 区域报告
  │  两份区域报告 → Commander LLM 融合 全局报告
  ▼
Step 5 — 全局报告输出
  "搜索完成。北部区域在 [43,-33] 发现疑似被困人员，
   南部区域未发现异常。建议派遣救援队前往北部标记点。"
```

**关键：每层的 LLM 只看到自己层面的信息。** UAV 不知道全局任务，Coordinator 不知道其他区域的情况，Commander 不知道每架 UAV 的具体飞行动作。信息在每一层被抽象和汇聚。

## 七、Gazebo 多机仿真

### 多机 Spawn 方案

PX4 SITL 支持多实例，每个实例用不同的 `PX4_SYS_AUTOSTART` 和 MAVLink 端口：

```bash
# UAV-A1: instance 0
PX4_SIM_MODEL=x500_lidar_2d_cam PX4_GZ_MODEL_POSE="0,0,0,0,0,0" \
  px4 -i 0   # MAVLink: 14540, SITL: 14560

# UAV-A2: instance 1
PX4_SIM_MODEL=x500_lidar_2d_cam PX4_GZ_MODEL_POSE="5,0,0,0,0,0" \
  px4 -i 1   # MAVLink: 14541, SITL: 14561

# UAV-B1: instance 2
PX4_SIM_MODEL=x500_lidar_2d_cam PX4_GZ_MODEL_POSE="0,5,0,0,0,0" \
  px4 -i 2   # MAVLink: 14542, SITL: 14562

# UAV-B2: instance 3
PX4_SIM_MODEL=x500_lidar_2d_cam PX4_GZ_MODEL_POSE="5,5,0,0,0,0" \
  px4 -i 3   # MAVLink: 14543, SITL: 14563
```

每架 UAV 的 AerialClaw Executor 通过不同的 MAVLink 端口连接对应的 PX4 实例。

### 端口分配

| 组件 | 端口 | 说明 |
|------|------|------|
| Gazebo | — | 共用一个 Gazebo 实例 |
| MicroXRCEAgent | 8888 | DDS Agent（共用） |
| PX4 instance 0 | 14540 / 14560 | UAV-A1 |
| PX4 instance 1 | 14541 / 14561 | UAV-A2 |
| PX4 instance 2 | 14542 / 14562 | UAV-B1 |
| PX4 instance 3 | 14543 / 14563 | UAV-B2 |
| Commander Web | 6000 | 主节点大屏 |
| Coordinator-A Web | 6100 | 子节点 A 面板 |
| Coordinator-B Web | 6101 | 子节点 B 面板 |
| Executor-A1 Web | 6200 | UAV-A1 控制台 |
| Executor-A2 Web | 6201 | UAV-A2 控制台 |
| Executor-B1 Web | 6210 | UAV-B1 控制台 |
| Executor-B2 Web | 6211 | UAV-B2 控制台 |

## 九、设计原则总结

1. **同一套代码，不同的灵魂** — 代码不绑定设备类型，身份文档定义角色
2. **自然语言贯穿全链路** — 主→子→机 全是自然语言，LLM 自己理解和拆解
3. **信息逐层抽象** — 每层只看到自己层面的信息，报告逐级汇聚
4. **内核与界面分离** — Agent 能力不依赖 Web 服务，界面按角色按需加载
5. **向下兼容** — v2.0 的所有单机能力完整保留，v3.0 是纯增量
